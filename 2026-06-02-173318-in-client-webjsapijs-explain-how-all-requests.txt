  How all requests travel over HTTPS

  1. BASE_URL = window.location.origin (line 17)

  window.location.origin returns the scheme + host + port of the page that
  loaded the script вҖ” for example https://whatsas.example.com. The scheme is
  whatever the browser used to fetch the page itself. If the page was served
  over HTTPS (which it must be in production вҖ” browsers enforce this for many
  Web Crypto APIs you use), then BASE_URL is https://вҖҰ automatically. No
  hardcoded string is needed.

  2. request() appends the path and calls fetch() (line 85)

  //                      ^^^^^^^^^^^^^^^^
  //  e.g. "https://whatsas.example.com/auth/login"

  4. What this means for your sensitive data

  - The JWT (sent as Authorization: Bearer вҖҰ on every authenticated call)

  How to fix it вҖ” two layers, pick both

  1. Client-side redirect (one line, in index.html or app.js):

  if (location.protocol === 'http:' && location.hostname !== 'localhost') {
      location.replace('https:' + location.href.slice(5));
  }

  This runs before any API call is made. The localhost carve-out lets local dev
  continue over HTTP.

  2. Server-side Strict-Transport-Security header:

  Once your server/gateway sends:
  Strict-Transport-Security: max-age=31536000; includeSubDomains
  the browser will refuse to make any HTTP request to that origin for a year вҖ”
  the redirect never even hits the network. This is the stronger guarantee and
  should be added to your server config regardless.

  Right now you are relying entirely on whatever redirect the gateway/server
  does. If that redirect is misconfigured or absent, the client will happily
  send credentials unencrypted. The client-side guard is a cheap safety net
  worth adding.
